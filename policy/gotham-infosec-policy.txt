# Gotham Ltd. Information Security Regulatory Policy

## SECTION 1: REGULATORY FRAMEWORK

1.1 Regulatory Mandate
    1.1.1 All personnel shall comply with this Information Security Policy without exception.
    1.1.2 Non-compliance shall result in disciplinary action up to and including termination of employment.

1.2 Normative References
    1.2.1 This policy shall be interpreted in conjunction with:
        a) NIST SP 800-53 Security Controls
        b) ISO 27001:2022 Information Security Standards
        c) Indonesia Personal Data Protection Act
        d) General Data Protection Regulation

## SECTION 2: ORGANIZATIONAL PROFILE

2.1 Organizational Identification
    2.1.1 The entity shall be identified as Gotham Ltd.
    2.1.2 Primary business activities shall be limited to automotive spare parts manufacturing.

2.2 Operational Jurisdiction
    2.2.1 The policy shall govern operations across:
        a) Headquarters in Jakarta
        b) Branch offices in Bandung and Surabaya
        c) Manufacturing facility in Bogor
        d) All digital and physical assets

## SECTION 3: RISK MANAGEMENT PROTOCOL

3.1 Risk Identification Directive
    3.1.1 Organizational risk assessments shall:
        a) Conduct comprehensive threat analyses bi-annually
        b) Document all identified potential vulnerabilities
        c) Develop mitigation strategies for each identified risk

3.2 Risk Classification Mechanism
    3.2.1 Risks shall be classified by:
        a) Probability of occurrence
        b) Potential financial impact
        c) Operational disruption potential

## SECTION 4: PERSONNEL SECURITY REGULATIONS

4.1 Recruitment Security Protocols
    4.1.1 All potential employees shall undergo:
        a) Comprehensive background verification
        b) Security clearance assessment
        c) Mandatory confidentiality agreement execution

4.2 Access Control Regulations
    4.2.1 System access shall be:
        a) Granted solely based on job-specific requirements
        b) Reviewed quarterly
        c) Immediately revoked upon employment termination

## SECTION 5: TECHNICAL SECURITY MANDATES

5.1 Network Protection Directive
    5.1.1 Network security shall mandate:
        a) Malicious packet detection threshold at 20% of total traffic
        b) Continuous monitoring intervals of 5 minutes
        c) Immediate automated response to threshold breaches

5.2 Encryption Compliance
    5.2.1 All communication channels shall:
        a) Implement TLS 1.3 minimum
        b) Use approved cryptographic cipher suites
        c) Undergo mandatory annual certificate rotation

## SECTION 6: INCIDENT RESPONSE PROTOCOL

6.1 Incident Classification Directive
    6.1.1 Security incidents shall be classified by:
        a) Severity level
        b) Potential organizational impact
        c) Required immediate response actions

6.2 Reporting Compliance
    6.2.1 Incident reporting shall adhere to strict timelines:
        a) Critical incidents: Reporting within 1 hour
        b) High-severity incidents: Reporting within 4 hours
        c) Medium-severity incidents: Reporting within 24 hours

## SECTION 7: ENFORCEMENT MECHANISMS

7.1 Compliance Enforcement
    7.1.1 Non-compliance consequences shall include:
        a) Formal written warnings
        b) Mandatory additional security training
        c) Potential employment termination

7.2 Audit Protocols
    7.2.1 Security assessments shall be conducted:
        a) Quarterly internal comprehensive audits
        b) Annual external compliance reviews
        c) Unannounced periodic spot checks


Section 4.1 - IoT Device Authentication Requirements

4.1.1 Password Management
- All default passwords must be changed before production deployment
- Minimum password complexity requirements:
  * 12 characters minimum length
  * Include uppercase, lowercase, numbers, and special characters
  * No dictionary words
  * No sequential patterns
- Passwords must be stored using secure hashing (bcrypt/Argon2)
- No hardcoded credentials in firmware or configuration

4.1.2 Authentication Mechanisms
- Implement certificate-based authentication where possible
- Enable two-factor authentication for administrative access
- Implement account lockout after failed attempts
- Regular password rotation requirement
- Password history enforcement

4.1.3 Access Control
- Implement role-based access control
- Principle of least privilege
- Regular access review and audit
- Automatic session timeout

Section 6.1 - API Security Requirements

6.1.1 Authentication & Authorization
- Implement OAuth 2.0 with PKCE for mobile apps
- Require strong JWT configuration:
  * Minimum RS256 algorithm
  * Required claims: iss, sub, aud, exp, iat, jti
  * Regular key rotation
- Implement role-based access control (RBAC)
- Enable MFA for sensitive operations

6.1.2 API Design & Implementation
- Versioning required for all APIs
- Rate limiting implementation mandatory
- Input validation on all parameters
- Sanitize all output data
- Implement proper error handling

6.1.3 Mobile App Security
- Certificate pinning required
- Secure local storage implementation
- Anti-tampering measures
- Secure logging practices


Section 3.1 - IoT Network Security Requirements

3.1.1 Certificate Management
- All devices must use valid certificates from approved CAs
- Certificate pinning required for critical devices
- Regular certificate rotation and revocation checks
- Minimum TLS version 1.3 required

3.1.2 Network Segmentation
- IoT devices must be isolated in separate VLAN
- Inter-VLAN routing strictly controlled
- Regular network access audits
- Network monitoring and IDS deployment

3.1.3 Protocol Security
- Secure MQTT (MQTT over TLS) required
- Client certificate authentication mandatory
- Strong cipher suite requirements
- Message signing for critical commands

3.1.4 Device Authentication
- MAC address filtering and validation
- Strong device authentication required
- Regular credential rotation
- Failed authentication monitoring

All IoT devices deployed within the organization must implement strong encryption for both data in transit and at rest. Specifically, any transmission of personally identifiable information (PII), including video feeds, sensor data, and user credentials, must use industry-standard encryption protocols such as TLS 1.2 or higher. Additionally, data stored on cloud servers or local devices must be encrypted using approved algorithms (e.g., AES-256). Devices must undergo regular security assessments and audits to verify compliance with these standards. Any identified vulnerabilities that expose sensitive data must be remediated within 30 days of discovery, and interim mitigation measures must be applied immediately to safeguard user privacy and comply with applicable data protection regulations.


Section 5.2 - Role-Based Access Control Requirements

5.2.1 Token Validation
- All authentication tokens must be cryptographically signed using RS256
- Tokens must include role information in JWT claims
- Token elevation requests must be validated against authorized role matrix

5.2.2 API Security
- Implement API rate limiting
- Require re-authentication for privilege elevation
- Implement path canonicalization
- Block directory traversal characters
- Log all privilege elevation attempts

5.2.3 Network Segmentation
- Segregate BMS admin interface to dedicated VLAN
- Implement jump box for administrative access
- Require VPN for remote management


Section 7.1 - Secure Update Requirements

7.1.1 Update Transport Security
- HTTPS required for all update communications
- Certificate pinning mandatory
- TLS 1.3 minimum requirement
- Regular certificate rotation

7.1.2 Firmware Integrity
- Code signing with strong keys required
- Minimum RSA-4096 or ECC P-384
- Secure boot implementation
- Firmware integrity verification
- Anti-rollback protection

7.1.3 Update Process
- Staged update deployment
- Automatic rollback capability
- Update authentication required
- Version control enforcement
- Update logging and monitoring
